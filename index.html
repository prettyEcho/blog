基本数据类型

标识符

标识符就是变量、对象、属性的名字。

标识符规范

  • 第一个字符必须是字母、下划线或美元符$
  • 其他字符可以是字母、数字、下划线或美元符$
  • 不能把关键字、保留字、true、false、null、Infinity、NaN、Undefined用做标识符

数据类型

  • 数字（Number）：整数和小数
  • 字符串（String）: 字符组成的文本
  • 布尔值（Boolean）: true和false
  • undefined ： 表示“未定义”或不存在，即由于目前没有定义，所以此处暂时没有任何值
  • null ：表示无值，即此处的值就是“无”的状态。
  • 对象（Object）: 各种值的集合
  • Symbol ：

通常，我们将数值、字符串、布尔值称为原始类型（primitive type）的值，即它们是最基本的数据类型，不能再细分了。而将对象称为合成类型（complex type）的值，因为一个对象往往是多个原始类型的值的合成，可以看作是一个存放各种值的容器。至于undefined和null，一般将它们看成两个特殊值。

对象分为：基本对象({})、数组、函数
JS对象可分为狭义对象（{}）、广义对象（JS将一切都视为对象）

判断类型

判断到底是什么类型，有三种方法：

  • typeof运算符
  • instanceof运算符
  • Object.prototype.toString方法
  1. typeof运算符

  • 原始类型
    字符串、数字、布尔值分别返回string、number、boolean

  • 对象

    函数、基本对象、数组分别返回 function、object、object

    typeof {}; // object

  • undefinded

    typeof undefined; // undefined

    可用于检测一个未定义的变量而不报错

  • null

    typeof null; // object

  • window
    在浏览器环境下： typeof window; // object
    在其他环境下： typeof window; // undefined

内存空间和上下文

堆与栈

堆，就像书架，你想找到什么书，只需要知道书名就可以了。
栈，先进后出，后进先出。

执行上下文

执行上下文存在于栈（函数调用栈）里面，最下面的永远是全局上下文，当函数被调用时，该函数上下文就会入栈，当函数执行完成时，函数上下文就会出栈。

单线程

当前js执行的永远是处于栈顶的上下文，下面的上下文只有等待。

同步执行

当前js执行的永远是处于栈顶的上下文，下面的上下文只有等待。

全局上下文只有一个，它在浏览器关闭时出栈。

变量对象

变量对象存在于当前执行上下文中，我们简单的可以理解为它存在于栈里面。
基础数据类型（undefined,null,String,Number,Boolean）直接存在于变量对象中，改变时，直接操纵它的值。
引用数据类型在变量对象中存在的时指针（地址），对象实际存在堆中，我们操控对象实际在操控其指针。

垃圾回收机制

当函数上下文出栈时，垃圾回收机制会回收其中的变量。全局的变量不会被清除，直到浏览器关闭，可以手动给变量赋值为null,使其被回收。

writable
true 当且仅当可能用 赋值运算符 改变与属性相关的值。默认为 false。

DOM基础

###什么是DOM？
DOM，顾名思义，文档对象模型，它是HTML和XML的编程接口。我们无法直接操纵HTML，而是通过DOM把文档解析为一个由节点和对象组成的结构对象。我们可以通过脚本或其他编程语言来操纵DOM，从而改变文档的结构，样式和内容。

###DOM节点分类？
DOM节点可用元素属性nodeType来分辨类型，分为1（元素节点）／2（属性节点）／3（文本节点）。

###创建节点
创建元素节点：document.createElement()
创建文本节点： document.createTextNode("Hello World!")

###获取节点
// 返回当前文档中第一个类名为 “myclass” 的元素
var el = document.querySelector(".myclass");

// 返回一个文档中所有的class为”note”或者 “alert”的div元素
var els = document.querySelectorAll("div.note, div.alert");

// 获取元素
var el = document.getElementById('xxx');
var els = document.getElementsByClassName('highlight');
var els = document.getElementsByTagName('td');

###获取节点关系
// 获取父元素、父节点
var parent = ele.parentElement;
var parent = ele.parentNode;

// 获取子节点,结果为一个节点数组，子节点可以是任何一种节点，可以通过nodeType来判断
var nodes = ele.children;

// 当前元素的第一个/最后一个子元素节点
var el = ele.firstElementChild;
var el = ele.lastElementChild;

// 下一个/上一个兄弟元素节点
var el = ele.nextElementSibling;
var el = ele.previousElementSibling;

/ 添加、删除子元素
ele.appendChild(el);
ele.removeChild(el);

// 替换子元素(el1替换el2)
ele.replaceChild(el1, el2);

// 插入子元素
parentElement.insertBefore(newElement, referenceElement);

###属性操作
// 获取一个{name, value}的数组
var attrs = el.attributes;

// 获取、设置属性
var c = el.getAttribute('class');
el.setAttribute('class', 'highlight');

// 判断、移除属性
el.hasAttribute('class');
el.removeAttribute('class');

// 是否有属性设置
el.hasAttributes();

由一个小问题引发的思考

前言

有一次在开发一个插件的时候，遇到了一个小小的问题，但是却很匪夷所思。

我们看：

请问，a的距页面顶部的距离是多少？

答：0。

不对。

What?

20px。

这是什么鬼？

我当时是蒙了，经过我一番折腾。发现了原因。

块状元素和行内元素

你真的理解这两个看似简单的概念吗？

我的理解是：

先说说盒模型，一个元素可以抽象成一个盒子，盒子是有由content、padding、border、margin构成。而盒子的大小是content+padding+border，margin决定其与其他元素的位置关系。

所以，margin是决定位置关系的。

我们比喻块元素是有型的东西，有大小，有形状。有型的东西之间当然是有位置关系的，他们由margin决定。但是行内元素无大小，它只是包含文本元素，或者行内元素。当行内元素中无内容时，它就变成了无形的东西，无形的东西想去哪就去哪，所以就不存在位置关系。

当我们给上面元素加上内容时，距离分分钟变成0，因为它有了形态。也就有了距离。

其实还有许多其他方法，只要使这个无形的东西变为有型的就行了，例如BFC。

客户端渲染VS服务器渲染

客户端渲染

基本模式

渲染过程

客户端发出请求 ➡️ 找到指定服务器上的html ➡️ 通过HTTP传到浏览器，进行构建、渲染 ➡️
下载JS、CSS文件 ➡️ 执行JS文件，通过ajax调取数据 ➡️ 接受数据，继续构建、渲染（部分页面） ➡️ 展示完整页面

优点及缺点

优点： 前后端彻底分离，各端人员可以各司其职，提高工作效率。
降低服务器压力。
缺点： 页面渲染速度慢（外网请求，响应较慢），用户体验性不好。
不利于SEO。

基本模式

  • 中间层

渲染过程

用户输入url ➡️ nginx改变指向到Controller ➡️ Controller指定到具体的vm页面 ➡️ 通过ajax调取数据，在服务器进行渲染生成html页面(内网速度快) ➡️ 将html页面通过http返回给浏览器 ➡️ 下载js/css ➡️ 加载js／css ➡️ 展示完整页面

优点及缺点

优点： 页面渲染速度快（），用户体验好。

前后端分离

选用哪种渲染模式

1.客户端渲染：很容易实现。
2.服务器端渲染：要求前端开发人员有服务器端开发语言相关知识。

优点

1.前后端分工明确，提高工作效率。
2.前后端耦合性低，可读性好，利于后期维护。
3.局部性能提高，前端可使用按需加载。

缺点

1.对前端开发人员要求较高。
2.需要有专门的服务器渲染的开发者。

前后端不分离

选用哪种渲染模式

1.客户端渲染：多此一举。
2.服务器端渲染：很容易实现。

优点

1.不需要有专门的服务器渲染的开发者。
2.前后端耦合性低，可读性好，利于后期维护。
3.局部性能提高，前端可使用按需加载。

缺点

1.前后端密切合作，增加时间、沟通成本，开发效率低。
2.前后端耦合性高，可读性不好，不利于后期维护。
